# 📋 Prerequisites

Before getting started, ensure you have:

- Node.js (Latest LTS recommended)
- npm
- Git
- A GitHub account
- Basic knowledge of React, TypeScript, and Vite

Check versions:

```bash
node -v
npm -v
git --version
```

---

# ⚙️ Project Setup

## 1. Fork the Repository

Click the **Fork** button on GitHub.

---

## 2. Clone your Fork

```bash
git clone https://github.com/<your-username>/SvaBharat.git
```

---

## 3. Navigate into the Project

```bash
cd SvaBharat
```

---

## 4. Install Dependencies

```bash
npm install
```

---

## 5. Start Development Server

```bash
npm run dev
```

---

## 6. Build the Project

```bash
npm run build
```

---

## 7. Run Linter

```bash
npm run lint
```

# 🌱 Development Workflow

1. Fork the repository.
2. Create a new branch.
3. Make your changes.
4. Test your changes.
5. Commit your work.
6. Push the branch.
7. Open a Pull Request.

---

# 🌿 Branch Naming Convention

Use meaningful branch names.

### Features

```text
feature/add-navbar
```

### Bug Fixes

```text
fix/login-error
```

### Documentation

```text
docs/update-readme
```

### Refactoring

```text
refactor/navbar-component
```

### Performance

```text
perf/improve-rendering
```

---

# 📝 Commit Message Guidelines

Follow meaningful commit messages.

Examples:

```text
feat: add responsive navbar

fix: resolve mobile layout issue

docs: update README

style: improve button spacing

refactor: simplify authentication logic

perf: optimize image loading
```

# 💻 Coding Standards

Please follow these best practices:

- Write clean and readable code.
- Use meaningful variable names.
- Keep components modular.
- Avoid duplicate code.
- Follow existing project structure.
- Remove unused imports.
- Format code before committing.
- Keep functions focused on a single responsibility.
- Use consistent indentation.
- Prefer reusable components whenever possible.

---

# 🧪 Testing Before Submitting

Before opening a Pull Request:

- Ensure the project builds successfully.
- Run the linter.
- Verify your changes work as expected.
- Ensure there are no console errors.
- Test responsive layouts if UI changes are involved.

---

# 🚀 Pull Request Process

Before creating a Pull Request:

- Sync your fork with the latest upstream changes.
- Create a dedicated branch.
- Make focused changes.
- Keep PRs small and easy to review.
- Update documentation if needed.

When opening your Pull Request:

- Provide a clear title.
- Describe what you changed.
- Reference related issues (e.g., `Closes #23`).
- Include screenshots or screen recordings for UI changes.
- Respond to review feedback promptly.

---

# 🐛 Reporting Issues

When creating an issue, include:

- A clear title.
- Description of the problem.
- Steps to reproduce.
- Expected behavior.
- Actual behavior.
- Screenshots (if applicable).
- Browser/OS information (if relevant).

Please check existing issues before creating a new one to avoid duplicates.

---

# ✅ Contributor Checklist

Before submitting your Pull Request, make sure:

- [ ] Code builds successfully.
- [ ] Linting passes.
- [ ] No unnecessary files are included.
- [ ] Documentation is updated if required.
- [ ] Commit messages follow project conventions.
- [ ] Your branch is up to date with the latest changes.
- [ ] UI changes include screenshots (if applicable).
- [ ] Pull Request has a clear description.
